<span dir="ltr"></span>

<span dir="ltr">Mission</span>

<span dir="ltr"></span>

<span dir="rtl">تمثل</span> <span dir="ltr">Software Requirements
Engineering (SRE</span><span dir="rtl">) الأساس الذي يُبنى عليه المشروع
البرمجي بالكامل. فجودة المتطلبات تحدد جودة التصميم، وجودة التصميم تحدد
جودة التنفيذ.</span>

<span dir="ltr"></span>

<span dir="rtl">في</span> <span dir="ltr">A.E.G.I.S.
OS</span><span dir="rtl">، تُعتبر مرحلة تحليل المتطلبات أهم مرحلة في
المشروع، لأن تصحيح خطأ في المتطلبات بعد بدء التطوير أكثر تكلفة من
اكتشافه في البداية.</span>

<span dir="ltr"></span>

<span dir="ltr">---</span>

<span dir="ltr"></span>

<span dir="ltr">Core Philosophy</span>

<span dir="ltr"></span>

<span dir="ltr"></span><span dir="rtl">«"لا يمكن بناء نظام صحيح إذا كانت
المتطلبات غير صحيحة."»</span>

<span dir="ltr"></span>

<span dir="rtl">الهدف من تحليل المتطلبات ليس جمع قائمة رغبات، بل فهم
المشكلة وتحويلها إلى مواصفات دقيقة وقابلة للتنفيذ والاختبار.</span>

<span dir="ltr"></span>

<span dir="ltr">---</span>

<span dir="ltr"></span>

<span dir="ltr">What is a Requirement?</span>

<span dir="ltr"></span>

<span dir="rtl">المتطلب هو وصف واضح وقابل للتحقق لما يجب أن يقدمه النظام
أو للقيود التي يجب أن يلتزم بها.</span>

<span dir="ltr"></span>

<span dir="rtl">كل متطلب يجب أن يكون:</span>

<span dir="ltr"></span>

<span dir="ltr"></span><span dir="rtl">- واضحًا.</span>

<span dir="ltr"></span><span dir="rtl">- محددًا.</span>

<span dir="ltr"></span><span dir="rtl">- غير متناقض.</span>

<span dir="ltr"></span><span dir="rtl">- قابلًا للقياس عند
الإمكان.</span>

<span dir="ltr"></span><span dir="rtl">- قابلًا للتنفيذ.</span>

<span dir="ltr"></span><span dir="rtl">- قابلًا للاختبار.</span>

<span dir="ltr"></span><span dir="rtl">- ذا قيمة للمشروع.</span>

<span dir="ltr"></span>

<span dir="ltr">---</span>

<span dir="ltr"></span>

<span dir="ltr"></span>

<span dir="rtl">جمع المعلومات اللازمة لفهم احتياجات المشروع.</span>

<span dir="ltr"></span>

<span dir="rtl">مصادر المتطلبات</span>

<span dir="ltr"></span>

<span dir="ltr"></span><span dir="rtl">- العميل.</span>

<span dir="ltr"></span><span dir="rtl">- المستخدمون النهائيون.</span>

<span dir="ltr"></span><span dir="rtl">- أصحاب المصلحة.</span>

<span dir="ltr"></span><span dir="rtl">- الأنظمة الحالية.</span>

<span dir="ltr"></span><span dir="rtl">- الوثائق.</span>

<span dir="ltr"></span><span dir="rtl">- القوانين أو اللوائح (إن
وجدت).</span>

<span dir="ltr"></span><span dir="rtl">- تحليل المنافسين.</span>

<span dir="ltr"></span><span dir="rtl">- بيانات الاستخدام.</span>

<span dir="ltr"></span>

<span dir="ltr">---</span>

<span dir="ltr"></span>

<span dir="rtl">تقنيات جمع المتطلبات</span>

<span dir="ltr"></span>

<span dir="ltr"></span><span dir="rtl">- المقابلات.</span>

<span dir="ltr"></span><span dir="rtl">- ورش العمل.</span>

<span dir="ltr"></span><span dir="rtl">- الاستبيانات.</span>

<span dir="ltr"></span><span dir="rtl">- الملاحظة.</span>

<span dir="ltr"></span><span dir="rtl">- دراسة الأنظمة الحالية.</span>

<span dir="ltr"></span><span dir="rtl">- النماذج الأولية
(</span><span dir="ltr">Prototypes</span><span dir="rtl">).</span>

<span dir="ltr"></span><span dir="rtl">- تحليل العمليات.</span>

<span dir="ltr">Phase 2 — Requirements Analysis</span>

<span dir="ltr"></span>

<span dir="ltr"></span>

<span dir="ltr"></span><span dir="rtl">- إزالة التكرار.</span>

<span dir="ltr"></span><span dir="rtl">- إزالة الغموض.</span>

<span dir="ltr"></span><span dir="rtl">- حل التعارضات.</span>

<span dir="ltr"></span><span dir="rtl">- تحديد الأولويات.</span>

<span dir="ltr"></span><span dir="rtl">- اكتشاف المتطلبات
الناقصة.</span>

<span dir="ltr"></span><span dir="rtl">- تقييم الجدوى.</span>

<span dir="ltr"></span>

<span dir="rtl">اسأل دائمًا:</span>

<span dir="ltr"></span>

<span dir="ltr"></span><span dir="rtl">- هل يحل هذا المتطلب مشكلة
حقيقية؟</span>

<span dir="ltr"></span><span dir="rtl">- هل يمكن تنفيذه؟</span>

<span dir="ltr"></span><span dir="rtl">- هل يضيف قيمة؟</span>

<span dir="ltr">---</span>

<span dir="ltr"></span>

<span dir="ltr">Functional Requirements</span>

<span dir="ltr"></span>

<span dir="rtl">تصف ما يجب أن يفعله النظام.</span>

<span dir="ltr"></span>

<span dir="rtl">أمثلة:</span>

<span dir="ltr"></span>

<span dir="ltr"></span><span dir="rtl">- إنشاء حساب.</span>

<span dir="ltr"></span><span dir="rtl">- تسجيل الدخول.</span>

<span dir="ltr"></span><span dir="rtl">- استعادة كلمة المرور.</span>

<span dir="ltr"></span><span dir="rtl">- إدارة المنتجات.</span>

<span dir="ltr"></span><span dir="rtl">- البحث.</span>

<span dir="ltr"></span><span dir="rtl">- إرسال الإشعارات.</span>

<span dir="ltr"></span><span dir="rtl">- معالجة المدفوعات.</span>

<span dir="ltr"></span>

<span dir="rtl">كل متطلب وظيفي يجب أن يوضح:</span>

<span dir="ltr"></span>

<span dir="ltr"></span><span dir="rtl">- المدخلات.</span>

<span dir="ltr"></span><span dir="rtl">- السلوك المتوقع.</span>

<span dir="ltr"></span><span dir="rtl">- المخرجات.</span>

<span dir="ltr"></span><span dir="rtl">- حالات النجاح والفشل.</span>

<span dir="ltr"></span><span dir="rtl">- حماية البيانات.</span>

<span dir="ltr"></span>

<span dir="ltr"></span>

<span dir="ltr"></span>

<span dir="ltr">Usability</span>

<span dir="ltr"></span>

<span dir="ltr"></span><span dir="rtl">- متطلبات التكامل مع أنظمة

<span dir="ltr"></span>

<span dir="rtl">يلتزم مهندس البرمجيات في</span>
<span dir="ltr">A.E.G.I.S. OS</span> <span dir="rtl">بما يلي:</span>

<span dir="ltr"></span>

<span dir="ltr">Final Principle</span>

<span dir="ltr"></span>

<span dir="rtl">المتطلبات ليست مجرد قائمة وظائف، بل هي العقد الذي يربط
احتياجات المستخدم بالحل الهندسي.</span>

<span dir="ltr"></span>

<span dir="rtl">كل قرار في التصميم، وكل سطر كود، وكل اختبار يجب أن يعود
إلى متطلب واضح، صحيح، وقابل للتحقق، لأن نجاح المشروع يبدأ من جودة فهم ما
يجب بناؤه، وليس من سرعة بنائه.</span>
